# 常见问题

## 1. 模块安装后还需要关心内部的 `lib`、`cpp` 或头文件吗

不需要。

安装模块后，直接在项目中引用 `火山.JadeView` 包即可。文档只关注：

- 类
- 方法
- 常量
- 事件
- 回调用法

## 2. 为什么有些事件不自动订阅

因为这类事件会影响默认行为或开销较大：

- `window-closing`
- `webview-will-navigate`
- `webview-new-window`
- `webview-download-started`
- `file-drop`

前四个属于可拦截事件，默认自动订阅会改变应用行为。

## 3. 可拦截事件应该返回什么

- 返回 `1`：拦截
- 返回 `0`：放行

当前只有以下事件解释返回值：

- `window-closing`
- `webview-will-navigate`
- `webview-new-window`
- `webview-download-started`

## 4. IPC 为什么不直接做成普通定义事件

因为 IPC 需要返回文本结果，而火山普通事件更适合整数返回值。

当前推荐：

- `置IPC频道回调(...)`
- `注册IPC通道(...)`

这样桥接层才能稳定地把火山文本返回成底层需要的文本结果。

## 5. IPC 推荐用静态回调还是非静态回调

推荐非静态回调，也就是对象方法。

推荐原因：

- 更容易访问对象状态
- 更适合按通道分发业务逻辑
- 更符合火山项目的组织方式

当前推荐写法是：

- `本对象`
- 输出名字符串

## 6. 异步对话框回调推荐怎么写

推荐对象方法回调，不必自己管理底层指针。

桥接层已经负责：

- 拷贝原始 JSON
- 调用 `jade_text_free(...)`
- 将稳定文本传回火山对象方法

## 7. 同步对话框为什么能直接返回文本

因为桥接层已经处理了：

1. 复制 SDK 返回的 UTF-8 文本
2. 调用 `jade_text_free(...)`
3. 转成火山 `文本型`

## 8. 托盘菜单为什么之前只能添加一个

根因通常是把临时字符串指针直接塞进原生菜单结构。

当前模块已经改成自持有菜单字符串和菜单项状态，因此可以稳定支持多个菜单项。

## 9. `tray-event` 或 `tray-menu-command` 是否支持拦截

不支持。

托盘事件属于通知类事件，返回值会被忽略。

## 10. 第二实例事件里为什么收到的是“参数数组”而不是原始 JSON

这是当前模块做的二次处理：

- 解析 `argv`
- 移除 `argv[0]` 可执行文件路径
- 将剩余命令行参数转成火山数组

这样业务层更容易直接使用。

## 11. 如何排查事件没有触发

建议按这个顺序检查：

1. 是否已经执行 `JadeView.初始化()`
2. 事件是否属于默认自动订阅列表
3. 若不是，是否已经手动调用 `订阅事件(...)`
4. 托盘事件是否已经先执行 `JadeView_托盘.创建()`
5. IPC 是否已经先执行 `置IPC频道回调(...)`

## 12. 文档应重点看什么

优先看：

1. `main.wsv`
2. 快速开始
3. `JadeView` 类说明
4. `JadeView_托盘` 类说明
5. 常见问题
